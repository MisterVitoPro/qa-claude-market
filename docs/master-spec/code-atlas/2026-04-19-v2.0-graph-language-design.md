# Code Atlas v2.0: Graph Language & Query Capability

**Date:** 2026-04-19  
**Status:** Design (awaiting approval)  
**Scope:** Add semantic graph representation (graph-schema.json) and queryable graph language with `/code-atlas:query` skill

---

## 1. Motivation

Code Atlas v1.2 generates accurate dependency data but stores it as raw JSON (`import_graph`, `importer_counts`). Claude must parse and reason about the graph structure manually, without semantic context about *why* connections matter.

v2.0 adds:
- **Semantic enrichment** â€” annotate modules and dependencies with roles, criticality, stability, test coverage, and impact information
- **Queryable representation** â€” `graph-schema.json` designed for Claude to ask structured questions about dependencies
- **Query skill** â€” `/code-atlas:query` lets Claude explore the graph without manual JSON parsing

This enables Claude to understand "how things work" at a higher level: which modules are critical, which changes ripple widely, which dependencies are optional vs. core.

---

## 2. Goals

- Generate semantic graph with enriched module and edge metadata (roles, criticality, stability, test coverage, impact)
- Provide queryable interface via JSON-based operations (dependencies, dependents, filters, transitive analysis)
- Keep graph focused on key modules only (from atlas.json's key_files + high_traffic) to bound size and relevance
- Support incremental updates when code changes
- Maintain backward compatibility with v1.2 (atlas.json and state.json unchanged)

---

## 3. Non-Goals

- Complex path-finding or pattern matching in v2.0 (deferred to v2.1)
- Circular dependency visualization (flagged in edges, not visualized)
- Symbol-level graph (classes, functions) â€” module/file granularity only
- Real-time graph updates during development (scan-based, not watch-based)

---

## 5. Schema: graph-schema.json

### 5.1 Structure

```json
{
  "_header": {
    "schema_version": 2,
    "plugin_version": "2.0.0",
    "generated_at": "<ISO 8601 UTC>",
    "baseline_commit": "<short SHA>",
    "scan_root": "."
  },
  "nodes": {
    "<module_or_file_path>": {
      "type": "module|file",
      "files": ["<path>", ...],
      "role": "entry_point|core_module|utility|config|middleware|model|etc.",
      "criticality": "critical|high|medium|low",
      "stability": "stable|evolving|experimental",
      "test_coverage": "well_tested|partial|untested",
      "description": "<one-line>"
    }
  },
  "edges": [
    {
      "source": "<path>",
      "target": "<path>",
      "type": "direct_import|dynamic_import|inheritance|composition|configuration|sideeffect",
      "strength": "core|utility|optional",
      "directionality": "required|circular|conditional",
      "impact": "breaking_change_risk|ripple_effect_magnitude|<none>"
    }
  ],
  "metadata": {
    "total_nodes": <int>,
    "total_edges": <int>,
    "key_modules_analyzed": <int>,
    "circular_dependency_count": <int>
  }
}
```

### 5.2 Node Types & Attributes

**Type: `module`**
- Group of related files (e.g., `src/auth/` â†’ contains `index.ts`, `middleware.ts`, `service.ts`)
- Represents logical subsystem
- Has a `files` array listing constituent files

**Type: `file`**
- Individual source file (included when referenced as standalone, or when explicitly queried)
- No `files` array
- Typically utility or singular-purpose files

**Node attributes:**

| Attribute | Values | How Determined |
|-----------|--------|-----------------|
| `role` | `entry_point`, `core_module`, `utility`, `config`, `middleware`, `model`, `public_api`, `internal` | Heuristics: entry points detected from directory name / importer count; core_module = top importers; utility = low criticality; inferred from structure |
| `criticality` | `critical`, `high`, `medium`, `low` | Blast radius: importer_count >= 10 â†’ critical; >= 5 â†’ high; >= 2 â†’ medium; else low |
| `stability` | `stable`, `evolving`, `experimental` | Convention inference: files matching `experimental/`, `beta/`, `dev/` â†’ experimental; high churn (recent changes) â†’ evolving; else stable |
| `test_coverage` | `well_tested`, `partial`, `untested` | Presence of `*.test.ts`, `*.spec.ts` sibling files: both â†’ well_tested; one â†’ partial; none â†’ untested |
| `description` | string | Directory name semantics + first docstring line if available; fallback to inferred purpose |

### 5.3 Edge Types & Attributes

**Edge attributes:**

| Attribute | Values | Meaning |
|-----------|--------|---------|
| `type` | `direct_import`, `dynamic_import`, `inheritance`, `composition`, `configuration`, `sideeffect` | Nature of the dependency: static vs. dynamic, structural vs. runtime |
| `strength` | `core`, `utility`, `optional` | Is this edge essential (breaks if removed), convenient, or conditional? |
| `directionality` | `required`, `circular`, `conditional` | Flow: required = A needs B; circular = A â†” B; conditional = A sometimes needs B |
| `impact` | `breaking_change_risk`, `ripple_effect_magnitude`, (empty) | Consequences if source changes: breaking changes if target's API shifts; ripple if many downstream users |

### 5.4 Scope: Key Modules Only

**graph-schema.json includes:**
- All modules/files in `atlas.json.key_files` (up to 15)
- All modules/files in `atlas.json.high_traffic` (up to 10)
- Their immediate dependencies and dependents (within the key set)

**Excludes:**
- Leaf utility files with no importers (unless in key_files)
- Third-party libraries (external only via annotation on edges)
- Test files, build artifacts, documentation

This keeps graph-schema.json bounded (typically < 50 KB) while covering the most important structural relationships.

## 6. Query Language & `/code-atlas:query` Skill

### 6.1 Query Format

Claude generates JSON query objects. Supported operations:

#### `dependencies_of`
```json
{
  "operation": "dependencies_of",
  "module": "src/auth"
}
```
**Returns:** All modules/files that `src/auth` imports (direct + transitive per max_depth)

#### `dependents_of`
```json
{
  "operation": "dependents_of",
  "module": "src/utils/logger"
}
```
**Returns:** All modules/files that import `src/utils/logger` (direct + transitive per max_depth)

#### `filter`
```json
{
  "operation": "filter",
  "conditions": {
    "criticality": "critical",
    "role": "core_module",
    "stability": "stable"
  }
}
```
**Returns:** All nodes matching ALL conditions (AND logic)

#### `transitive_dependents`
```json
{
  "operation": "transitive_dependents",
  "module": "src/models",
  "max_depth": 3
}
```
**Returns:** All nodes reachable from `src/models` following edges inbound, up to N hops (default 2, max 5)

### 6.2 Skill Behavior

**Invocation:**
```
/code-atlas:query

{
  "operation": "dependents_of",
  "module": "src/auth"
}
```

**Response:**
```json
{
  "query": { "operation": "dependents_of", "module": "src/auth" },
  "matched_nodes": [
    {
      "id": "src/api",
      "type": "module",
      "role": "core_module",
      "criticality": "critical",
      "stability": "stable",
      "description": "HTTP endpoints"
    }
  ],
  "matched_edges": [
    {
      "source": "src/api",
      "target": "src/auth",
      "type": "direct_import",
      "strength": "core",
      "impact": "breaking_change_risk"
    }
  ],
  "summary": "Found 3 modules depending on src/auth, 5 edges total"
}
```

**Error handling:**
- Missing `graph-schema.json` â†’ suggest running `/code-atlas:map`
- Invalid query format â†’ return schema hint with valid operations
- Module not in graph â†’ return empty result with note "module not in key set or is external"
- Circular dependency â†’ edges marked with `directionality: "circular"`, traversal continues

### 6.3 Query Execution Algorithm

**Filter operation:**
1. Iterate all nodes
2. Check each condition against node attributes
3. Collect matching nodes
4. Collect all edges where source OR target is in matched nodes
5. Return both sets

**Dependencies/Dependents (transitive):**
1. Locate source node in graph
2. BFS/DFS from source, following edges in appropriate direction
3. Stop at max_depth
4. Collect all visited nodes + edges between them
5. Return subgraph

## 7. Workflow: Generation & Update

### 7.1 `/code-atlas:map` (Full Scan)

**New steps after existing import extraction:**

1. **Semantic analysis for each key module:**
   - Determine `role`: entry points by name heuristic, high importers â†’ core_module, others by importer count
   - Assign `criticality`: from importer_counts (critical â‰Ą 10, high â‰Ą 5, medium â‰Ą 2, else low)
   - Infer `stability`: check directory names (`experimental/`, `beta/`, `dev/`) and recent change frequency
   - Measure `test_coverage`: count `*.test.ts`, `*.spec.ts` files colocated
   - Synthesize `description`: directory name semantics + first docstring if found

2. **Edge annotation:**
   - For each edge in import_graph within key set:
     - Infer `type`: analyze import statement (static â†’ direct_import, dynamic â†’ dynamic_import, etc.)
     - Assign `strength`: is this import used in core logic (core) or convenience (utility)? Optional if conditional?
     - Set `directionality`: does target depend on source (circular)? Is import conditional?
     - Mark `impact`: if source is critical and target is high-traffic, mark breaking_change_risk

3. **Write graph-schema.json:**
   - Assemble nodes + edges
   - Populate metadata counts
   - Write to `.code-atlas/graph-schema.json`

### 7.2 `/code-atlas:update` (Incremental)

**Existing strategy selection (micro/targeted/full) applies to graph-schema as well:**

| Strategy | Work |
|----------|------|
| Micro (< 5 changed files) | Re-analyze changed files only, patch edges, rewrite graph-schema |
| Targeted (5-30 changed files) | Micro work + re-analyze structural changes, merge graph |
| Full (> 30 changes or 3+ new dirs) | Regenerate entire graph-schema via `/map` pipeline |

**Implementation:**
- Track which key modules changed (from hash-diff)
- Re-analyze only those modules for semantic metadata
- Re-extract imports for affected modules
- Update edges + metadata counts
- Rewrite graph-schema.json

## 9. Testing Strategy

**Unit tests:**
- Query parsing: valid/invalid JSON, missing fields
- Filter operation: node matching logic, edge collection
- Transitive traversal: BFS correctness, max_depth boundary, cycle handling
- Edge cases: empty result sets, single-node graphs, unreachable nodes

**Integration tests:**
- Run `/code-atlas:map` on reference repo (e.g., Code Atlas itself)
- Validate graph-schema structure against schema
- Cross-check: all nodes in graph exist in import_graph; all edges have valid source/target
- Query smoke tests: known questions about Code Atlas structure answered correctly

**Manual validation:**
- Use `/code-atlas:query` to answer questions like "what modules depend on src/utils/logger?"
- Compare results against `atlas.json.high_traffic` and dependency_flow
- Verify criticality assignments match intuition (high-traffic modules are critical)

## 10. Risks & Tradeoffs

1. **Heuristic-based semantic inference.** Roles, criticality, stability are inferred, not explicit. Mitigation: annotations are bounded (small set of discrete values) and testable against intuition.

2. **Key-set only scope.** Graph doesn't cover entire codebase. Mitigation: key set is curated by existing atlas agents; users can request full graph via `/code-atlas:query` with explicit filter if needed (v2.1 feature).

3. **Import extraction is approximate.** Dynamic imports, macro-generated imports, runtime requires missed. Mitigation: same caveat as v1.2; edges are best-effort.

4. **Circular dependencies in queries.** BFS could infinite-loop on cycles. Mitigation: track visited nodes, max_depth prevents runaway traversal.

---

## 11. Out of Scope (v2.1+)

- Path-finding: "what's the dependency chain from A to B?"
- Pattern detection: "find all modules following dependency injection pattern"
- Impact analysis: "if I change X, which other modules could break?"
- Full-codebase graph: only key modules in v2.0
- Real-time graph updates: scan-based only
